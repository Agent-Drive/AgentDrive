# Reducto PDF Parsing Design

## Problem

Docling PDF parsing is broken in production. The `DocumentConverter` downloads layout models from Hugging Face at runtime, and Cloud Run's shared IP gets rate-limited (429). PDFs have never successfully processed on the deployed server. Additionally, Docling pulls heavy dependencies (PyTorch, HF Hub, OpenCV) that bloat the Docker image and cause cold start issues.

## Solution

Replace Docling with Reducto's cloud API for PDF parsing. Reducto handles OCR, layout detection, and table extraction, returning clean markdown that feeds into the existing `MarkdownChunker`. This is a small change with no modifications to the chunking hierarchy, enrichment, or embedding pipeline.

## Design

### 1. PDF Chunker â€” Reducto Integration

Replace the Docling call in `PdfChunker.chunk_bytes()` with a Reducto API call:

```
chunk_bytes(data, filename, metadata)
  â†’ extract signed_url from metadata (generated by ingest.py)
  â†’ Reducto client.parse.run(input=signed_url)
  â†’ concatenate result chunk contents into single markdown string
  â†’ MarkdownChunker.chunk(markdown, filename)
```

The Reducto client is synchronous (`reducto.Reducto`). `chunk_bytes` is a sync method called synchronously from `process_file` â€” same as Docling today. No `asyncio.to_thread` needed since the entire `process_file` already runs in a queue worker task.

**Reducto parse configuration:**
- `formatting.table_output_format`: `"dynamic"` (markdown for simple tables, HTML for complex)
- `settings.ocr_system`: `"standard"` (1 credit/page)
- No Reducto-side chunking configuration needed â€” we concatenate all returned content into a single markdown string and let our `MarkdownChunker` handle segmentation

**Error handling:** Remove the existing `try/except: return []` that silently swallows errors. Let exceptions propagate up to `process_file()` which already handles failures (rollback + set status=FAILED). The queue worker adds a 15-minute timeout on top. The Reducto SDK has built-in retries (2x with exponential backoff) for transient errors (429, 5xx).

**No temp file needed:** Docling required writing bytes to a temp file. Reducto fetches via signed URL. The temp file write and cleanup code is removed.

### 2. Signed URL Generation â€” in ingest.py, not the chunker

The signed URL is generated in `ingest.py` (where `StorageService` already exists) and passed to the chunker via `metadata["signed_url"]`. This keeps chunkers as pure data transformers with no infrastructure dependencies.

**GCS signed URL on Cloud Run:** Application Default Credentials on Cloud Run use `compute_engine.Credentials` which lacks `sign_bytes` for V4 signing. Two options:

- **Option A (recommended):** Use the IAM signBlob API via `google.auth.iam.Signer`. Requires granting the Cloud Run service account the `iam.serviceAccountTokenCreator` role on itself.
- **Option B (fallback):** Skip signed URLs. Upload file bytes directly to Reducto via their SDK (`client.upload(file=data)`), then parse using the returned `file_id`. Simpler but uploads the file twice (once to GCS, once to Reducto).

The implementation should try Option A first. If IAM signing proves problematic, fall back to Option B.

Add a new method to `StorageService`:

```python
def generate_signed_url(self, gcs_path: str, expiration_minutes: int = 30) -> str:
    bucket = self._client.bucket(settings.gcs_bucket)
    blob = bucket.blob(gcs_path)
    return blob.generate_signed_url(
        version="v4",
        expiration=timedelta(minutes=expiration_minutes),
        method="GET",
    )
```

### 3. Changes to ingest.py

Generate the signed URL and pass it via metadata:

```python
# Current
chunk_groups = chunker.chunk_bytes(data, file.filename)

# Proposed
storage = StorageService()
signed_url = storage.generate_signed_url(file.gcs_path)
chunk_groups = chunker.chunk_bytes(
    data, file.filename,
    metadata={"gcs_path": file.gcs_path, "signed_url": signed_url},
)
```

Non-PDF chunkers ignore `metadata`. No ripple effects. The `data` bytes are still downloaded because non-PDF chunkers need them â€” the signed URL is only used by `PdfChunker`.

Note: `StorageService` is already instantiated in `process_file` (line 30) for the download call, so we reuse that instance.

### 4. Config + Dependencies

**`config.py`:** Add `reducto_api_key: str = ""`. Env var: `REDUCTO_API_KEY`.

**`pyproject.toml`:** Swap `docling` for `reductoai>=0.16.0,<1.0`. Massive dependency reduction â€” Docling pulls PyTorch, HF Hub, model weights, OpenCV. Reducto is a thin `httpx` wrapper.

**Cloud Run:** Add `REDUCTO_API_KEY` to service secrets.

**Dockerfile:** Remove Docling-specific system deps if any (`libgl1`, etc.). Check during implementation.

## Testing

- Empty result: Reducto returns no chunks â†’ returns `[]`
- API error: Reducto raises exception â†’ propagates (not swallowed)
- Verify chunker reads `metadata["signed_url"]` and passes to Reducto

**Existing tests:** No changes to `test_files.py` (mocks `enqueue`). For `test_ingest.py`, add a mock for the Reducto client at the `pdf.py` module level, similar to how enrichment/embedding are mocked in `conftest.py`. Verify no existing test path reaches `PdfChunker.chunk_bytes` without mocking.

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Scope | PDF parsing only | Other chunkers work fine; Reducto complements, not replaces |
| File transfer | GCS signed URL (with direct upload fallback) | File already in GCS; avoids double upload. Fallback if IAM signing is problematic. |
| Signed URL generation | In `ingest.py`, not chunker | Keeps chunkers as pure data transformers; avoids coupling to StorageService |
| Reducto chunking | Ignore / concatenate | Our MarkdownChunker builds parent-child hierarchy; Reducto chunks are flat |
| Table format | `dynamic` | Auto-switches markdown/HTML based on table complexity |
| OCR mode | `standard` | 1 credit/page; `agentic` (2x cost) available for complex docs later |
| Docling | Remove entirely | Broken in production, heavy deps, no value as fallback |
| Error handling | Let exceptions propagate; remove silent catch | Existing pipeline handles failures; SDK retries transient errors |
| Async | Stays synchronous | `chunk_bytes` is sync, called from sync context in `process_file`. Same as Docling. |
| SDK version | `>=0.16.0,<1.0` | Pin to current major to avoid breaking API changes |
